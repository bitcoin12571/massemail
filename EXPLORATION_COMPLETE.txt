## EXPLORATION SUMMARY

### 1. DELIVERY STATUS PAGE ✓
- Component: QueueMonitor.jsx (/frontend/src/components/)
- Navigation: Dashboard.jsx page 3
- Title: "Starea livrărilor" (Romanian) / "Delivery status" (English)
- Stats: 4 cards showing waiting, sending now, delivered, failed

### 2. BACKEND QUEUE SYSTEM ✓
- Service: queueService.js (/backend/src/services/)
- Data: In-memory jobs array + stats object
- Processing: 50 emails in parallel batches
- API Routes: campaigns.js (/backend/src/routes/)
  - GET /campaigns/stats/queue
  - POST /campaigns/stats/queue/clear
  - POST /campaigns/stats/queue/retry

### 3. DATA MODELS ✓
- Email: persisted in database (9 status values)
- Campaign: persisted in database (5 status values)
- Contact: persisted in database (personalization data)

### 4. DATA PERSISTENCE ✓
- Job Queue: IN-MEMORY (lost on restart)
- Stats: IN-MEMORY (lost on restart)
- Email Status: DATABASE (always saved)
- User Prefs: localStorage (browser)

### 5. KEY CHARACTERISTICS ✓
- Auto-refresh: 2.5 seconds (default)
- Processing: Immediate (not deferred)
- Batch size: 50 emails at a time
- Failure handling: Store reason, mark as failed, allow retry
- Database: SQLite (dev) or PostgreSQL (prod)

### 6. LIMITATIONS IDENTIFIED ✓
- Queue lost on restart
- Stats reset on restart
- Single-server only
- No job history
- No scheduling support
